## 📋 八阶段实施计划

### 阶段 1：数据库 Schema & 核心类型定义
**目标**：建立游戏世界的数据底座

**需要创建的 Schema 表**：
- `scenes` — 场景节点（id, parentId, name, type, description, properties, exits）
- `entities` — 实体（id, name, type, sceneId）
- `components` — 组件存储（id, entityId, type, data）
- `events` — 事件日志（Event Sourcing 核心）
- `snapshots` — 世界快照（简化版）
- `agent_states` — Agent 当前状态

**需要创建的核心类型**：
- `src/lib/server/game/types.ts` — UUID、Scene、Entity、Component 联合类型、ActionIntent、Event 等

---

### 阶段 2：世界内核（World Kernel）
**目标**：实现唯一真相源

**模块**：
1. **`entity-store.ts`** — ECS 实体/组件存储
   - `createEntity(name, type)`
   - `addComponent(entityId, component)`
   - `getComponent(entityId, type)`
   - `queryEntities(sceneId?, componentType?)`

2. **`scene-tree.ts`** — Scene 树管理
   - `createScene(name, type, parentId?)`
   - `moveEntity(entityId, targetSceneId)`
   - `getScenePath(sceneId)` → `青河镇/幸福小区/3号楼/302室/卧室`
   - `getEntitiesInScene(sceneId)`

3. **`world-kernel.ts`** — 世界内核（所有状态变更的唯一入口）
   - 维护当前世界时间
   - 维护场景树引用
   - 维护 ECS Store 引用
   - `dispatch(action)` — 所有变更走这里
   - `getState()` — 只读获取完整世界状态
   - `getAgentSubView(agentId)` — 获取 Agent 的感知子视图

---

### 阶段 3：事件系统 & 调度器
**目标**：实现不固定 Tick 的时间推进

**模块**：
1. **`event-types.ts`** — 核心事件类型定义
   - `AGENT_DECIDE`, `AGENT_ACTION`, `SCENE_CHANGE`, `SPEAK`, `TIME_ADVANCE`

2. **`event-bus.ts`** — 内存事件总线
   - `emit(event)` — 同步/异步事件分发
   - `on(eventType, handler)`
   - 所有事件都写入 `events` 表做溯源

3. **`scheduler.ts`** — 不固定 Tick 调度器（最小堆实现）
   - `schedule(atTime, event)` — 按仿真时间插入事件
   - `tick()` — 取出当前时间之前的所有事件，排序执行
   - `advanceTime(seconds)` — LLM 推断跳到下一个有意义时刻
   - `getCurrentTime()` — 当前仿真时间

4. **`validator.ts`** — 行动合法性校验（简化版）
   - 检查 Agent 是否在某场景中
   - 检查目标场景是否可达
   - 检查行动是否合法（比如不能穿墙）

---

### 阶段 4：LLM 客户端层
**目标**：让内核能调用 LLM

**模块**：
1. **`client.ts`** — 通用 LLM 客户端（OpenAI 兼容格式）
   - `chatCompletion(messages, modelConfig)` → 返回 `{ content, usage }`
   - 支持流式输出（为后续 SSE 预留）
   - **本次 MVP 只支持单模型**，配置从环境变量读取

2. **`prompt-builder.ts`** — Prompt 组装器
   - `buildAgentPrompt(agentState, worldState, memories)` → System + User messages
   - 定义一个标准 JSON Schema 让 LLM 输出 ActionIntent：
     ```json
     {
       "thought": "思考过程",
       "action": { "type": "SPEAK|MOVE|INTERACT|WAIT", "target": "...", "content": "..." },
       "timeAdvanceSeconds": 300
     }
     ```

3. **`stream-handler.ts`** — 流式输出处理（简化版，为 SSE 预留接口）

---

### 阶段 5：Agent 系统
**目标**：实现 3 个生活在同一房间的 Agent

**模块**：
1. **`agent.ts`** — Agent 基类
   - `id, name, personality, currentSceneId`
   - `perceive(worldState)` → 构建感知上下文
   - `decide()` → 调用 LLM 获取 ActionIntent
   - `execute(action)` → 通过 WorldKernel 执行

2. **`player-agent.ts`** — Player Agent（LLM 全权控制）
   - 继承 Agent，decide() 调用主交互 LLM
   - 输出完整的思考 + 行动

3. **`npc-agent.ts`** — NPC Agent（简化版规则 + 随机 LLM 触发）
   - **本次 MVP 用极简规则**：每轮随机选择 speak/move/wait
   - 预留 LLM 接口但暂不调用（V0.2 再接入）

---

### 阶段 6：持久化 & 事件溯源
**目标**：世界状态可变更有迹可循

**模块**：
1. **`event-log.ts`** — 事件日志写入
   - 每个事件保存到 SQLite `events` 表
   - 格式：`{ id, worldTime, type, agentId, data, createdAt }`

2. **`snapshot-manager.ts`** — 简化版快照
   - `saveSnapshot()` → 将当前 worldState 序列化到 `snapshots` 表
   - 本次 MVP 只支持手动 save/restore，不实现 fork/revert

---

### 阶段 7：最小化 API 层
**目标**：让前端能连接内核

**端点**：
1. **`/api/world/[worldId]/sse/+server.ts`** — SSE 实时流

   - 事件格式：`world:update`, `agent:action`, `system:log`

2. **`/api/world/[worldId]/command/+server.ts`** — 上帝指令

   - `POST { command: "pause" }` → 暂停

   - `POST { command: "reset" }` → 重置世界
