## 一、愿景与核心设计哲学

| 维度           | 设计选择                                                        |
| -------------- | --------------------------------------------------------------- |
| **世界观**     | 多层级 Scene 树（小镇 → 建筑 → 房间 → 物品），无限嵌套          |
| **时间**       | **不固定 Tick** — 由 LLM/事件驱动跳跃，而非固定节拍             |
| **AI 策略**    | **按需唤醒**：主角(LLM全权)，NPC(规则为主+随机LLM介入)          |
| **架构核心**   | **确定性引擎 + Specialist LLM 编排**，引擎掌握世界权威          |
| **状态持久化** | SQLite 是当前状态 + Event Sourcing 做历史 + Snapshot 做版本控制 |
| **代码安全**   | DSL JSON 规则为主，QuickJS 沙箱为辅，编译验证+code review       |
| **通信**       | 自研轻量 Event Bus，A2A 语义 + MCP 垂直工具集成                 |
| **前端**       | Svelte 5 Runes + SvelteKit 2 + Tailwind v4 + SSE 流式输出       |
| **LLM 接入**   | **LiteLLM 通用适配层**，JSON 配置多模型路由，按需调用           |

### 1.1 已确认的关键决策

| #   | 决策项           | 结论                                                                                                                                                                         |
| --- | ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **LLM 接入方式** | 使用 LiteLLM 作为统一适配层，通过 JSON 配置各模型 endpoint、token、名称及场景路由。先只支持 OpenAI 类型接口（`/chat/completions`），保留扩展性。Token 需要节省但不是首要约束 |
| 2   | **动态代码执行** | 简单规则用 DSL JSON（precondition → effect），复杂逻辑用 QuickJS 沙箱。LLM 可信但需防幻觉，简单场景用编译验证+code review，复杂代码走沙箱                                    |
| 3   | **真人介入方式** | **A+B+C 全部支持**：A-附身操控角色、B-fork副本聊天交互（可继续/回退）、C-上帝模式与系统LLM对话，由LLM以合理方式操控世界                                                      |
| 4   | **版本控制粒度** | 整个游戏世界级别 fork/revert。每次 fork 是一个完整的世界快照分支，可独立演进                                                                                                 |
| 5   | **前端展示模式** | **面板模式为主**：顶部显示世界状态/场景状态/Player 状态，下方多个可切换面板显示 Player 的交互/思考/系统日志/LLM调用日志                                                      |

## 三、关键技术点详细设计

### 3.1 世界模型：Scene-Entity-Component-Item (SECI)

```typescript
// 核心类型定义
type UUID = string;
type ScenePath = string; // 例如: "青河镇/幸福小区/3号楼/302室/卧室"

interface Scene {
  id: UUID;
  parentId: UUID | null;        // 父场景，null 为根
  name: string;
  type: 'town' | 'building' | 'room' | 'outdoor' | 'abstract';
  description: string;          // LLM 可动态生成/更新
  properties: Record<string, any>; // 面积、朝向等
  children: UUID[];             // 子场景
  occupants: UUID[];            // 当前在此场景的实体
  items: UUID[];                // 场景中的物品
  exits: { direction: string; targetSceneId: UUID; condition?: string }[];
}

interface Entity {
  id: UUID;
  name: string;
  type: 'player' | 'npc' | 'item' | 'building';
  components: ComponentMap;
  // 不直接存位置，通过 Position 组件关联
}

type Component =
  | { type: 'Position'; sceneId: UUID; relX: number; relY: number; facing: string }
  | { type: 'Stats'; energy: number; maxEnergy: number; health: number; ... }
  | { type: 'Inventory'; capacity: number; items: UUID[] }
  | { type: 'Schedule'; dailyPlan: ScheduleEntry[] }
  | { type: 'Memory'; shortTerm: MemoryChunk[]; longTerm: MemoryChunk[] }
  | { type: 'Personality'; traits: Record<string, number>; voice: string }
  | { type: 'Relation'; targets: RelationEntry[] }
  | { type: 'Perception'; visibleEntities: UUID[]; audibleEvents: string[] };
```

### 3.2 不固定 Tick 的时间推进机制

```
┌─────────────────────┐
│   当前仿真时间 T     │
└──────────┬──────────┘
           │
    ┌──────▼──────┐
    │  检查事件队列  │ ───空──→ LLM 推断「下一次有意义时刻」
    │ (MinHeap)    │           → 直接跳到该时刻
    └──────┬──────┘
           │ 有事件
           ▼
    ┌──────────────┐
    │ 取出 T 时刻所有事件 │
    │ 按 ID 排序执行    │
    └──────┬─────────┘
           │
           ▼
    ┌──────────────┐
    │ 事件执行可能产生 │
    │ 新事件 → 插入队列 │
    └──────┬─────────┘
           │
           ▼
    ┌──────────────┐
    │ LLM 评估「中间  │
    │ 是否会有新事件」 │
    └──────┬─────────┘
           │
           ▼
    ┌──────────────┐
    │ 推入新事件     │
    │ 更新状态       │
    │ 记录事件日志   │
    └──────────────┘
```

**关键设计**：不是 "每隔 X 秒推进一次"，而是：

> **"世界在有事情发生时才前进，无事发生时 LLM 决定跳到下一个有意义时刻"**

LLM 的时间推理输出：

```json
{
	"timeAdvanceSeconds": 1800,
	"reason": "角色正在睡觉，期间无重要事件，跳到闹钟响",
	"scheduledEvents": [{ "atOffset": 1800, "type": "alarm", "entityId": "xxx", "description": "手机闹钟响" }],
	"granularity": "coarse",
	"nextDecisionPointReason": "闹钟响后角色需要决策是否起床"
}
```

### 3.3 多 LLM 分工与通信协议

每个角色的一轮行动触发一个 `ActionDAG`：

```
         ┌─────────────┐
         │  Agent 感知  │
         └──────┬──────┘
                │
         ┌──────▼──────┐     ┌─────────────┐
         │ 主交互LLM   │────→│ 记忆LLM检索  │ (并行)
         │ (思考+决策)  │     └─────────────┘
         └──────┬──────┘
                │
      ┌─────────┼─────────┐
      ▼         ▼         ▼
 ┌────────┐ ┌────────┐ ┌────────┐
 │ 叙事LLM │ │ 时间LLM │ │ 代码LLM │  (并行判定)
 │ 合法性  │ │ 推进推断 │ │ 规则生成 │  (按需唤醒)
 └────────┘ └────────┘ └────────┘
      │         │         │
      └─────────┴─────────┘
                │
         ┌──────▼──────┐
         │  编排器校验   │
         │  合并更新     │
         └──────┬──────┘
                │
         ┌──────▼──────┐
         │  世界状态更新  │
         │  写事件日志   │
         │  推前端流    │
         └─────────────┘
```

**共享上下文机制（Context by Reference）**：

- 所有 LLM 不直接传递世界状态的完整文本
- 传递 `contextRef: { worldStateHash, agentSubViewPaths, lastModified }`
- 编排器在调用 LLM 前按需组装 prompt

### 3.4 版本控制系统（Fork/Revert）

基于 Event Sourcing + Snapshot 的轻量级 Git-like 层：

```
tick 0 ──tick 100──tick 200──tick 300──tick 400──tick 500 (main分支)
   │        │         │         │
   │        │      snapshot   snapshot
   │        │      (每100tick)
   └────────┴──────────────────────────────→ fork "experiment" @ tick 300
                                    │
                                 tick 301 (新分支)
                                    │
                                 tick 400 (experiment)
```

**核心操作**：

- `fork(branchId, atTick)` → 从 snapshot + 日志重放，创建新分支
- `revert(branchId, toTick)` → 截断事件日志，重建状态
- `cherryPick(fromBranch, tickRange, toBranch)` → 提取事件应用到目标分支
- 自动每 N tick 生成 snapshot（后台异步）

### 3.5 动态代码执行安全策略

三层安全架构：

| 层级                            | 用途                         | 技术                                                      | 安全策略                          |
| ------------------------------- | ---------------------------- | --------------------------------------------------------- | --------------------------------- |
| **L1: DSL 规则** (默认)         | 节日活动、物品交互、状态变更 | JSON precondition → effect，引擎解释执行                  | ✅ 完美安全，LLM 只生成结构化数据 |
| **L2: 受限表达式**              | 数值公式、条件判断           | AST Evaluator（仅允许 `+ - * / > < ==`，无函数/全局访问） | ✅ 编译时验证，无代码执行风险     |
| **L3: QuickJS 沙箱** (复杂场景) | 动态节日脚本、特殊交互逻辑   | `quickjs-emscripten`，内存限制 50MB，超时 5s              | ⚠️ 沙箱隔离+白名单 API+审计日志   |

**安全原则**：

- LLM 可信但需防幻觉，简单场景（L1/L2）用编译验证+code review
- 复杂代码（L3）走 QuickJS 沙箱，不直接执行 node vm
- 所有 LLM 生成的代码/规则都记录审计日志，可追溯

暴露给沙箱的受控 API：

```typescript
{
  getState: (path: string) => any,           // 只读访问世界状态
  setState: (path: string, value: any) => void, // 受限写入（引擎二次校验）
  emitEvent: (type: string, data: any) => void, // 触发事件
  getTime: () => number,                     // 当前仿真时间
  log: (message: string) => void,            // 审计日志
}
```

### 3.6 LLM 模型路由与配置系统（LiteLLM 适配层）

通过 JSON 配置文件定义可用模型和路由策略：

```json
{
	"models": [
		{
			"id": "local-deepseek",
			"name": "DeepSeek-R1 满血版",
			"provider": "openai-compatible",
			"apiBase": "http://localhost:8000/v1",
			"apiKey": "sk-xxx",
			"model": "deepseek-ai/DeepSeek-R1",
			"maxTokens": 8192,
			"temperature": 0.7
		},
		{
			"id": "local-qwen",
			"name": "Qwen2.5-72B",
			"provider": "openai-compatible",
			"apiBase": "http://localhost:8001/v1",
			"apiKey": "sk-yyy",
			"model": "Qwen/Qwen2.5-72B-Instruct",
			"maxTokens": 4096,
			"temperature": 0.8
		}
	],
	"routing": {
		"default": "local-deepseek",
		"rules": [
			{ "task": "main-interaction", "model": "local-deepseek", "priority": 1 },
			{ "task": "memory-encode", "model": "local-deepseek", "priority": 2 },
			{ "task": "narrative-judge", "model": "local-deepseek", "priority": 2 },
			{ "task": "time-infer", "model": "local-qwen", "priority": 3 },
			{ "task": "code-generate", "model": "local-deepseek", "priority": 1 }
		]
	},
	"fallback": {
		"enabled": true,
		"fallbackModel": "local-qwen",
		"retryCount": 2
	}
}
```

**路由决策流程**：

1. Conductor 根据当前任务类型（`main-interaction`/`memory-encode`/...）查询路由规则
2. 选择对应模型，发起 OpenAI 格式请求
3. 失败时按 fallback 策略重试
4. 所有调用记录 token 消耗和延迟，用于后续优化

**扩展性预留**：

- `provider` 字段支持 `"openai-compatible" | "anthropic" | "google" | "custom"`
- 自定义 provider 只需实现 `LLMProvider` 接口即可接入

### 3.7 真人介入系统（HumanProxy）

三种介入模式统一通过 `HumanProxy` 处理：

#### 模式 A：附身（Possession）

```
玩家点击"附身" → 选择一个 Player/NPC → HumanProxy 接管该 Agent 的 decide() 方法
→ 玩家的前端输入直接作为该角色的 ActionIntent → 经过同样的 Validator → 执行
→ 其他 Agent 和 LLM 无法区分这是真人还是 LLM 控制
```

- 附身期间，该角色的感知仍然通过正常渠道获取（能看到什么、听到什么）
- 玩家通过聊天框发送消息，HumanProxy 将其转为 `ActionIntent { type: 'SPEAK', content: '...' }`
- 解除附身后，LLM 无缝接管

#### 模式 B：副本交互（Sandbox Chat）

```
玩家选择场景 + 角色 → 创建该场景的临时副本（内存分支，不写入主存档）
→ 玩家以"隐形角色"与目标角色对话
→ 可随时"继续"（将对话结果同步回主世界）或"回退"（丢弃副本）
```

- 副本是轻量级的内存分支，不触发 Snapshot 和 Event Log 写入
- 适合测试不同对话选项的后果
- "继续"时，副本中的关键记忆/关系变更可以合并回主世界（需玩家确认）

#### 模式 C：上帝模式对话（God Conversation）

```
玩家以"上帝"身份与 System LLM 对话
→ "我希望待会儿下雨"
→ System LLM 评估合理性 → 生成合理的事件序列（如"乌云聚集→风起→下雨"）
→ 推入 EventQueue，按时间逐步执行
→ 所有变更都通过正常事件系统，可被 fork/revert
```

- System LLM 有权生成世界级别的事件和规则变更
- 但所有变更都经过 WorldKernel 的合法性校验
- 上帝指令同样记录 Event Log，确保可追溯

**面板类型**（可动态添加/关闭）：

- **交互日志**：角色对话和行动，按时间线排列
- **思考流**：LLM 的思考过程（think 字段），可选择性显示/隐藏
- **系统日志**：Tick 推进、事件触发、规则执行、LLM 调用记录
- **LLM 调用详情**：完整 Prompt、Response、Token 消耗、耗时
- **场景树**：可展开/折叠的 Scene 树，点击节点查看详情
- **角色详情**：选中角色的完整属性、记忆、关系网

**模式切换**：

- 默认：观察模式（只看）
- 附身模式：底部输入框变为"以 [角色名] 的身份说/做..."
- 副本模式：显示"副本"标识，操作不影响主世界
- 上帝模式：底部输入框变为"我希望..."，System LLM 处理

### 优先级说明

1. **先做内核后做界面**：先让 tick 循环和 Agent 决策跑起来，哪怕没有前端
2. **先做主角后做 NPC**：先实现 Player Agent 的完整 LLM 交互，再实现 NPC 的规则调度
3. **先做单模型后做路由**：先用一个模型跑通全流程，再加入多模型路由
4. **先 DSL 后 QuickJS**：先用 JSON 规则满足大部分需求，复杂场景再引入沙箱

## 七、设计原则与约束

### 7.1 核心设计原则

1. **引擎至上**：WorldKernel 是唯一真相源，LLM 只有建议权，没有修改权
2. **事件驱动**：世界状态只能通过事件变更，所有变更可追溯
3. **按需唤醒**：不是每个 Agent 每轮都调 LLM，日常行为走规则，关键时刻才唤醒
4. **文本优先**：世界状态以文本为中心，AI 和玩家都通过文本交互，图形是后期扩展
5. **涌现叙事**：不预设剧情，让 AI 交互自然产生故事
6. **安全分层**：简单用 DSL，复杂用沙箱，绝不直接 eval LLM 生成的代码

### 7.2 技术约束

| 约束项          | 要求                                             |
| --------------- | ------------------------------------------------ |
| **单机运行**    | 不依赖外部服务（除 LLM API），SQLite 单文件存储  |
| **OpenAI 兼容** | LLM 客户端先只支持 OpenAI 格式，保留扩展接口     |
| **Token 意识**  | 记录每次调用的 token 消耗，支持用量统计和预警    |
| **版本控制**    | 整个世界可 fork/revert，分支独立演进             |
| **真人无痕**    | 真人介入对 Agent 不可见，接口与 LLM 控制完全一致 |

### 7.3 扩展预留

| 扩展点       | 当前方案          | 未来可扩展至                        |
| ------------ | ----------------- | ----------------------------------- |
| LLM Provider | OpenAI-compatible | Anthropic / Google / Azure / 自定义 |
| 代码沙箱     | QuickJS           | WASM / isolated-vm / 外部容器       |
| 存储         | SQLite            | PostgreSQL / Turso / 分布式         |
| 前端         | SSE 文字流        | WebSocket / 图形渲染 (Phaser)       |
| 部署         | 单机              | 服务端 multiplayer                  |

## 八、附录

### A. 术语表

| 术语               | 定义                                                       |
| ------------------ | ---------------------------------------------------------- |
| **Tick**           | 仿真时间的最小推进单位，但长度不固定                       |
| **Scene**          | 世界中的场景节点，可嵌套（小镇→建筑→房间）                 |
| **Agent**          | 有自主决策能力的实体（Player/NPC）                         |
| **ECS**            | Entity-Component-System，实体组件系统架构                  |
| **DSL**            | Domain Specific Language，领域特定语言（这里是 JSON 规则） |
| **Event Sourcing** | 只存储事件流，状态通过重放事件重建                         |
| **Snapshot**       | 某时刻的完整世界状态副本，用于加速恢复                     |
| **Fork**           | 从某时刻创建新的世界分支，可独立演进                       |
| **Revert**         | 回退到某时刻的状态，丢弃之后的历史                         |
| **SSE**            | Server-Sent Events，服务端单向推送技术                     |

- **LlamaBrain**：确定性权威架构
